# Standard Incident Report Template

## Incident ID

`INC-YYYYMMDD-001`

## Date

YYYY-MM-DD

## Service

Name the affected service, workload, cloud resource, or platform component.

## Severity

- Severity:
- Reason:
- Escalation criteria:

## Executive Summary

Briefly summarize what happened, which service was affected, and the current or expected user impact.

## Root Cause

Explain the confirmed or most likely root cause. If the root cause is not confirmed, state what is known and what still needs investigation.

## Impact

Describe the technical, user, business, or operational impact.

Include details such as:

- Affected users or systems
- Failed requests, jobs, or deployments
- Duration of impact
- Scope of degradation or outage

## Resolution

Document the actions taken to resolve or mitigate the incident.

Include:

- Immediate remediation steps
- Commands or configuration changes used, if relevant
- Validation steps used to confirm recovery
- Current status

## Prevention

List follow-up actions to reduce the chance of recurrence.

Examples:

- Monitoring or alerting improvements
- Runbook updates
- Configuration changes
- Infrastructure changes
- Testing improvements
- Deployment checks

## Lessons Learned

Summarize what the team learned from the incident.

Include:

- What worked well
- What slowed down investigation or recovery
- What should be improved before the next incident

## Notes

Add links to related logs, dashboards, pull requests, tickets, or reports.
